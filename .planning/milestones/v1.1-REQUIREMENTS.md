# Requirements: Threa TTRPG Assistant

**Defined:** 2026-01-26
**Core Value:** Players and Game Masters can easily access the system, manage their content securely, and focus on gameplay rather than administration.

## v1.1 Requirements

Requirements for v1.1 User Management & Authentication milestone. Each maps to roadmap phases.

### Registration & Authentication

- [x] **AUTH-01**: New user can self-register with username (unique), password (min 6 chars), and secret question/answer
- [x] **AUTH-02**: First registered user automatically receives Admin role
- [x] **AUTH-03**: Subsequent registered users receive User role by default
- [x] **AUTH-04**: User can initiate password reset by providing username
- [x] **AUTH-05**: System validates user identity using secret question/answer (case-insensitive, trimmed)
- [x] **AUTH-06**: User can set new password after successful secret answer validation

### User Management (Admin)

- [x] **USER-01**: Admin can view list of all users with username, display name, roles, and enabled status
- [x] **USER-02**: Admin can disable a user account (user locked out, data preserved)
- [x] **USER-03**: Admin can enable a previously disabled user account
- [x] **USER-04**: Admin can assign roles to users (User, GameMaster, Admin)
- [x] **USER-05**: Admin can remove roles from users
- [x] **USER-06**: Disabled users cannot log in

### User Profile

- [x] **PROF-01**: User can set a display name (shown in UI, separate from login username)
- [x] **PROF-02**: User can optionally provide an email address
- [x] **PROF-03**: User can update their email address at any time
- [x] **PROF-04**: User avatar displays via Gravatar when email is provided
- [x] **PROF-05**: User avatar displays initials fallback when email is not provided
- [x] **PROF-06**: User can view their own profile
- [x] **PROF-07**: Users can view other users' public profiles (display name, avatar)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Email verification for registration | No email sending capability, not required for semi-private deployment |
| Email-based password recovery | No email sending capability, using secret Q&A instead |
| Account deletion by user | Data retention preferred, admin can disable accounts |
| Multiple secret questions | One Q&A pair sufficient for semi-private deployment |
| CAPTCHA on registration | Semi-private deployment, known user group, low spam risk |
| Password history/reuse prevention | Unnecessary complexity for current scale |
